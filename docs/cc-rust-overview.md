# cc-rust（Lite）项目总览、架构与开发指南

本文旨在为团队成员提供对 cc-rust（Lite 版本）的全面理解与操作指南，包括架构设计、构建与运行方式、模块细解、开发流程、测试策略、部署要点以及未来演进路径。cc-rust 是在原版 Claude Code 的基础上，针对 Rust 生态与本地化需求，提供的一个精简而可扩展的实现。它保留最小可用核心，同时保留技能系统，以支持个性化工作流与长期演进。本文的内容面向开发者、测试人员、运维以及产品团队，力求避免模糊表述，给出可操作的细则与示例。

1. 引言

cc-rust（Lite）以 Rust 为核心实现语言，目标在于提供高效、可维护、可扩展的对话式代理框架。它实现了一套从请求接收、状态管理、到工具执行、结果汇聚的完整工作流，同时保留对外部服务的接入能力（Anthropic、OpenAI、Google 等后端 API 的 IPC/网关等实现子集在 rust/ api 模块中），以支持多提供商的替代方案。Lite 版本的核心定位是提供稳定的核心能力、易于本地化测试、并可按需扩展技能与工具集合。 

2. 架构总览

2.1 总体架构
- 后端核心：Rust 实现的 QueryEngine、生命周期管理、异步流式查询、工具执行与状态压缩等核心能力。
- 工具系统：28 种工具的定义与执行入口，支持扩展、拆分与组合使用。
- 技能系统：内置技能集与用户自定义技能的加载、注册、执行与记忆。
- 压缩管线：上下文压缩与缓存策略，确保对话历史在可控范围内保持高效、可预测的成本。
- 配置与权限：权限模型、配置管理、会话持久化、键/证书管理等关键能力。
- IPC 与前端：前端通过 headless 模式与 Rust 后端以 JSONL 协议通信，当前终端界面（OpenTUI）以 Bun/JavaScript 实现，形成清晰的前后端分离。

2.2 关键数据流
- 请求进入：前端或外部客户端发起对话请求，进入 Rust 的入口点 main.rs，进入 QueryEngine 的消息处理循环。
- 消息处理：通过 submit_message、deps、lifecycle 等模块组合实现消息的分解、依赖解析与状态转移。
- 工具执行：根据对话上下文与策略，调度相应工具执行、返回中间结果。
- 类型化与序列化：使用固定的类型系统确保跨模块的互操作性，输出格式遵循 JSONL/JSON 的传输协议。
- 压缩与记忆：在对话深度增加时，进入 compact 管道，对上下文进行压缩与记忆优化，控制资源成本。

2.3 安全性与隔离
- 路径隔离：cf. Path Isolation (Critical) 指引，确保全局数据、配置、技能等在不同系统间隔离，避免数据混淆或泄露。
- 权限模型：对工具的调用进行粒度控制，默认为 ask/auto/bypass 三种模式，保护风险操作。
- 错误与健壮性：实现稳健的错误处理、重试策略、回退机制，避免单点故障驱动整套系统崩溃。

3. 设计原则

3.1 最小可用、可扩展性并重
Lite 版本聚焦于核心能力，但预留扩展点，用于未来拓展技能、工具、外部适配器与多模态输入。

3.2 清晰的接口、明确的契约
模块之间采用清晰的输入输出契约，确保插件式扩展时的替换性与可测试性。

3.3 安全第一，默认保守
工具执行与外部 API 调用都具备权限判定、输入校验与错误边界，避免任意命令执行带来潜在风险。

3.4 可观测性与可操作性
对关键事件提供可观测性组件，便于追踪、调试与容量规划。

4. 构建与运行

4.1 依赖与环境
- 后端：Rust 编译器及相关工具链，目标：release 构建。
- 前端：OpenTUI 作为当前 UI，运行于 bun/Node 环境。
- 数据目录：遵循 Path Isolation 规定，避免跨环境污染。

4.2 构建步骤
- 后端构建：cargo build --release
- 前端构建：进入 `ui/` 目录执行 `bun install` 与 `bun run dev`，启动 OpenTUI 前端开发环境。
- 启动顺序：先启动后端（headless 模式可用于无 UI 的自动化场景），再启动前端 UI，前后端通过 headless IPC 进行通信。

4.3 运行与调试
- 日志：在后端关键节点输出结构化日志（JSON/文本），前端同样输出 UI 层日志。
- 调试入口：通过 --headless 与调试模式切换，利用 IPC 协议定位问题。
- 测试策略：单元测试覆盖工具执行、集成测试覆盖对话流、端到端测试覆盖 UI/CLI 入口。

4.4 清理与部署
- 产物：release 构建产物放置在 target/release/ 下。
- 部署：后端服务与前端 UI 打包分发，提供本地化的持久化路径。

5. 详细模块解析

5.1 后端（rust/src）主要目录与职责
- main.rs：应用入口，负责阶段性生命周期管理（Phase A/B/I），支持 --headless 标志，便于自动化测试与无 UI 场景。
- types/：核心类型定义，确保不同模块之间的契约一致。
- engine/：QueryEngine 以及系统提示词管理，负责对话上下文与策略决策。
- engine/lifecycle/：QueryEngine 生命周期的具体实现，包括 mod、types、submit_message、deps、helpers。
- query/：异步流式查询循环的实现，包含 loop_impl 与 loop_helpers，支持分块读取、流式输出与中间结果处理。
- tools/：28 种工具的实现及入口，覆盖搜索、文件、网络、系统命令等常见场景。
- skills/：技能系统，内置技能与用户自定义技能的加载、执行、持久化记忆能力。
- compact/：上下文压缩管道，用于对历史对话进行抽取、去冗与选择性记忆。
- commands/：28 条斜杠命令的实现，提供命令行交互接口与管理能力。
- api/：API 客户端集合，涵盖 Anthropic、OpenAI、Google 等后端接口适配层。
- auth/：认证与凭证管理，支持 API Key 与 Keychain 的集成。
- permissions/：权限系统，针对工具调用实施细粒度的访问控制。
- config/：配置管理模块，支持本地配置、远端配置和环境变量的覆盖规则。
- session/：会话持久化，确保对话状态可恢复与跨会话的上下文传递。
- ipc/：IPC 协议与 headless 通信实现，JSONL 流式传输格式。
- ui/：TUI 界面实现，提供命令行界面的显示、输入与交互。
- utils/：工具函数集合，提供通用的辅助能力。
- shutdown.rs：优雅关机实现，确保热更新及资源清理。

5.2 前端（ui/）与 IPC
- OpenTUI：当前终端渲染库，定义 UI 组件、输入框、消息列表等，通过 headless IPC 与 Rust 后端解耦。
- ui/src/ipc/client.ts：IPC 客户端，负责启动后端进程、发送/接收 JSONL 协议消息。
- ui/src/ipc/protocol.ts：协议定义，确保前后端对消息格式的统一理解。

5.3 架构与数据流在架构文档中的对应关系
- architecture/ink-terminal-frontend.md 保留为终端前端演进历史说明。
- 重要的跨模块契约文档应存储在 docs/ 目录的相关文档中，确保新人能快速上手。

5.4 新增服务模块与生态扩展
- services/ 目录下的新服务模块包括 tool_use_summary、session_memory、prompt_suggestion、lsp_lifecycle 等，用于支持更丰富的工作流与记忆能力。这些模块往往以服务化的方式暴露 API，便于其他工具或技能复用。

5.5 身份验证与凭证管理
- ApiClient::from_auth 的流程建立在环境变量检测与系统 Keychain 的解析之上，能够在多平台（macOS、Linux、Windows）上实现无痛的凭证管理。

6. 工具系统与权限模型

6.1 Tool 的契约
- 每个 Tool 实现一个固定接口，包含 name、description、inputJSONSchema、call()；并实现 isEnabled()，用于功能开关和干净的构建。
- Tools 应提供清晰的实现边界，避免跨工具的副作用。

6.2 权限模型
- ToolPermissionContext 提供三种工作模式：默认（ask）、auto、bypass。默认模式下遇到高风险操作需要用户确认，auto 模式下自动放行，bypass 状态下对指定工具跳过权限检查。
- 权限控制颗粒度可在工具级别进行配置，确保对核心操作的保护。

6.3 安全实践
- 为输入参数进行严格验证，避免注入攻击与异常输入。
- 对外部 API 调用进行超时、重试、熔断处理，避免单点故障、资源耗尽。
- 日志记录尽量结构化，便于监控与安全审计。

9. 部署与运行环境

9.1 本地开发与生产环境的差异
- 本地开发更强调快捷调试，生产环境强调稳定性、可观测性与容量规划。

9.2 数据持久化与备份
- 路径隔离要求本地持久化数据存放在专用目录，定期备份、日志轮转和清理策略需要在生产环境中明确。

9.3 容量规划
- 对话历史、上下文记忆、工具缓存等数据具有可观测的增长趋势，需定期评估存储和内存资源。

10. 监控、性能与容量规划

10.1 指标与日志
- 记录关键事件、工具执行耗时、上下文压缩效率、失败率、重试次数等。

10.2 性能优化点
- 优化查询循环的延迟、工具执行的并发控制、上下文压缩的算法选择。

10.3 容量规划策略
- 根据对话吞吐与平均上下文长度，估算内存占用，设置合理的缓存上限。

11. 常见问题排查

- 构建失败：请检查 Rust toolchain 是否齐全，是否存在依赖冲突。
- IPC 通信异常：请确认前后端的协议对齐、JSONL 行为是否符合预期，检查日志输出。
- 数据持久化问题：确认数据目录权限、磁盘空间及路径隔离策略是否正确。
- 权限拒绝：核对 ToolPermissionContext 设置与当前环境的调用场景。
